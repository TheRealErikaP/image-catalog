How to use the Image Catalog

A reference for you, your team, or anyone you hand this off to. You can reopen this anytime.

What it does Setting up a client Adding Wix images Adding GitHub images Push to GitHub Resources tab Bulk import AI descriptions Duplicates Using the export Tips & gotchas Handing this off
The short version: This tool keeps a running list of every image on a client's website — whether it's hosted on Wix or on GitHub — along with what each image is of and where it's used. When you start a new Claude chat to build a page, you paste the exported catalog and Claude knows exactly which images to use instead of making up placeholders.

1 What this tool actually does

For each client, you build a catalog — a running list of their images with URLs, descriptions, and tags. Two types of images go in:

  • Wix-hosted images that already exist on the client's site. You copy the URL from Wix and paste it here with a description so you (and Claude) can find it later.
  • New images hosted on GitHub Pages — drop a file, get a clean URL, then commit the file to the client's GitHub repo. Use these when you're adding fresh images to a site or when you want naming control.

The payoff: when you export the catalog, you get a clean reference Claude can read. No more hunting through Wix Media Manager for URLs mid-build.

2 Setting up a client

  1. Click + New client in the sidebar.
  2. Fill in:
    • Client name — required. Used as the workspace label.
    • Wix site URL — optional, for reference. Included in the export so Claude knows what site this is for.
    • GitHub user — defaults to therealerikap. Change only if you're hosting on a different account.
    • GitHub repo — the repo where new images will live. One repo per client is the cleanest setup (e.g., fds-assets, mylearningfarm-assets). Required if you want to add GitHub-hosted images to this client.
    • Notes — anything Claude should know. Brand palette, style guidelines, image conventions. This shows up in every export.
  3. Click Save client. You can edit these later by clicking edit next to the client name.
One-time GitHub setup per client: Create the repo on GitHub (public). Go to Settings → Pages → set source to main branch, root folder. Wait ~30 seconds. The site is now live at https://therealerikap.github.io/[repo-name]/. Any image file you push to that repo will be accessible at that URL.

3 Adding existing Wix images to the catalog

Use this when you need to catalog images the client already has on their Wix site.

  1. Go to the client's live Wix site or Wix editor.
  2. Right-click the image you want to catalog → Copy image address.
  3. In the Image Catalog, make sure you're on the + Add existing Wix image tab.
  4. Paste the URL into the Wix image URL field. A preview should appear.
  5. Fill in:
    • Name — something short and memorable. "Homepage hero," "About page portrait," "Services CTA background."
    • Description — what's actually in the image. This is what Claude reads to know when to use it. Be specific: "Exterior shot of commercial metal fire door in an office building, natural lighting" beats "fire door photo."
    • Tags — keywords for searching. Comma-separated.
    • Used on pages — so you know where it already appears. Helps avoid using the same image twice.
    • Alt text — for accessibility and SEO. If the Wix image has alt text, copy it here.
  6. Click Add to catalog.

4 Adding new images (GitHub-hosted)

Use this when adding a brand new image to a client's site, especially when you want predictable URLs.

  1. Switch to the + Add new GitHub image tab.
  2. Drop your image file onto the dropzone (or click to browse).
  3. The tool auto-generates a cleaned-up filename (lowercase, hyphens, no spaces). You can edit it before saving.
  4. Fill in the same description, tags, pages, and alt text fields as before.
  5. Click Add to catalog.
  6. Don't forget this step: go to the client's GitHub repo and upload the image file using the exact filename shown. The URL won't work until you do this.
    • Easiest way: on GitHub, open the repo → click Add file → Upload files → drag in → commit.
Heads up: The tool doesn't upload to GitHub for you — that's intentional (avoids storing credentials). If you skip the GitHub commit step, the catalog entry will exist but the URL will return a 404.

5 Using the export — the whole point of the tool

Once a client has a catalog, click Export for Claude in the client header.

Option A: Fast & disposable
  1. Hit Copy to clipboard.
  2. Open a new Claude chat.
  3. Paste it at the start of the chat with a message like: "Here's the FDS image catalog. I'm building a new services page — use images from this list."
  4. Claude now knows every available image and will reference them by URL.
Option B: Reusable (recommended for ongoing clients)
  1. Hit Download file.
  2. Upload [client]-image-catalog.md to the client's GitHub repo.
  3. In any future Claude chat, just say: "Pull the image catalog from https://therealerikap.github.io/fds-assets/[client]-image-catalog.md"
  4. Claude fetches the live file. You only update one place — re-export and re-upload whenever you add new images.
Markdown vs JSON: Use Markdown by default. It's readable by both humans and Claude, and it'll render properly when you view it on GitHub. JSON is there only if you want to pipe the catalog into other automation later.

6 Push to GitHub — upload files directly from here

You don't have to upload image files to GitHub separately anymore. If you've set a GitHub Personal Access Token in Settings, the + Add new GitHub image tab has a ⬆ Push to GitHub button that uploads the file for you.

  1. One-time: Go to github.com → Settings → Developer settings → Personal access tokens → Tokens (classic) → Generate new (classic). Check the repo scope. Copy the token.
  2. In this tool, click Settings & AI keys in the sidebar, paste the token into the GitHub PAT field, Save.
  3. Pick a client that has a GitHub user + repo configured (click edit next to the client name if not).
  4. Go to + Add new GitHub image, drop a file, fill in the metadata, click ⬆ Push to GitHub.
  5. The file uploads to the client's repo. If a file with that name already exists, it replaces it (useful for updating flyers). Then click Add to catalog.
Security note: Your PAT is stored in this browser's localStorage only. One token is used across all clients. If your browser gets compromised, revoke the token at GitHub — everything stops working cleanly with no lingering access.

7 Resources tab — catalog non-image URLs

Not everything you track for a client is an image. The Resources & links tab (toggle at the top of each client workspace) is for everything else: Calendly links, Google Forms, Drive folders, tracking pixels, dashboards, payment links, social profiles, landing pages, analytics, whatever.

  1. Switch to Resources & links view at the top of the client workspace.
  2. Paste a URL. The type auto-detects (Google Form, Calendly, Stripe, etc.) but you can override.
  3. Add a name, description, what pages it's used on, and tags — same fields as images.
  4. Use + Bulk import to dump a whole list at once — one URL per line, optional | name separator.

Resources show up in the Claude export too, grouped by type, so when you start a new Claude chat you can say "what's this client's Calendly link?" and Claude will know.

Use case examples: Tracking every Google Form a client uses for lead capture. Keeping Calendly links for different team members. Logging which Drive folders clients gave you access to. Storing Stripe payment links for different services. Knowing which landing page URL goes with which campaign.

8 Bulk import — paste a whole list at once

If you already have a list of image URLs (from a spreadsheet, a doc, an old export, whatever), use the + Bulk import links tab instead of adding one at a time.

  1. Click + Bulk import links tab in any client's add-image area.
  2. Paste URLs, one per line. Optionally add a description with | or - :
    • https://example.com/img.jpg
    • https://example.com/logo.png | Company logo used in footer
    • https://example.com/team.jpg - Team photo from 2024
  3. Click Parse & preview. You'll see every row with its thumbnail, duplicates flagged, and editable name/description fields.
  4. If you want AI to fill in descriptions for the empty rows, click ✨ Generate descriptions for empty rows. Costs roughly 1-2¢ per image.
  5. Click Add all to catalog. Duplicates get replaced, new ones get added.

9 AI descriptions (optional)

If you drop an Anthropic API key in Settings, every add form gets an ✨ Describe with AI button. Claude reads the image and writes the description, alt text, and tags for you.

  • No key set? Buttons still work — they fall back to a cleaned-up version of the filename.
  • Vision fails? If Wix blocks the image from CORS fetching (rare but possible on older Wix sites), you'll see a toast and it falls back to the filename. You can always paste a description manually.
  • Your key is private. It lives in this browser's localStorage only, and is only sent directly to Anthropic — never stored on any server.
  • Cost: Roughly 1-2¢ per image. A full bulk import of 100 images is about $1-2. You'll get a confirmation prompt before any bulk run.

10 Duplicate detection

Every time you add an image, the tool checks if you already have it. Three checks happen:

  1. Exact URL match — instant, free.
  2. Filename match — catches re-uploads of the same filename (e.g. updating a flyer).
  3. Byte-identical hash — catches re-uploads of the same file even if renamed.

When a duplicate is caught, you'll see a modal showing both side-by-side. You can:

  • Replace existing — overwrites the old entry with your new one. Good for "I updated the flyer with new dates."
  • Keep both — adds it as a new entry. Good for "this is a different image that happens to have the same filename."
  • Cancel — don't add anything.
  • 🔍 Compare with Claude — if you're unsure, Claude Vision looks at both images and tells you if they're identical, an updated version, or actually different. Uses one API call.
The honest limit: This catches byte-identical files, same URLs, and same filenames. It can't catch "same photo but a different JPG compression" or "same flyer resaved" unless you run Compare with Claude manually.

11 Tips, gotchas, and habits

  • Write descriptions for Claude, not for yourself. "Hero image" means nothing to Claude. "Wide-angle photo of a commercial metal fire door with push bar, natural light, used as hero on the homepage" means everything.
  • Update the "used on" field. This is how you avoid putting the same hero image on three different pages.
  • Re-export after changes. The catalog in your browser updates instantly, but if you use Option B above, you need to re-download and re-upload the markdown file to GitHub for Claude to see the changes.
  • The catalog lives in your browser. It's stored in this browser only — so use the same computer + browser, or export as JSON periodically as a backup. If you clear your browser data, the catalog is gone (though the GitHub repo copies survive).
  • Broken thumbnails aren't broken entries. If a Wix URL's preview fails (some are CDN-restricted), the URL is still saved correctly. Test the URL in a new browser tab to confirm.

12 Handing this off to a VA or teammate

If someone else is going to maintain the catalog:

  1. Send them the HTML file. They save it somewhere and open it in a browser (Chrome, Edge, Firefox, Safari — any modern browser works).
  2. Walk them through one client setup so they see how the Wix right-click-copy workflow goes.
  3. Make sure they understand: adding to the catalog is not the same as uploading to GitHub. For new images, both steps need to happen.
  4. Give them access to your GitHub account (or have them push to their own, and update the GitHub user field per client).
  5. Agree on a cadence — probably re-export + re-upload the markdown whenever images change, or weekly if catalogs change often.

They can reference this exact instructions panel anytime by clicking How to use this tool in the sidebar.
